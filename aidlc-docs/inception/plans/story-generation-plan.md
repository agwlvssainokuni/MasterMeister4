## ストーリー分割アプローチの選択肢

| アプローチ | 説明 | このプロジェクトでの適性 |
|---|---|---|
| Feature-Based | requirements.md §5.xの機能単位でエピックを構成 | 要件がすでに機能単位で整理済みのため親和性が高い |
| User Journey-Based | 「ユーザ登録から承認まで」等、一連の操作フローでストーリーを構成 | ユーザ登録フロー等一部機能には有効だが、クエリビルダー等の単機能には過剰 |
| Persona-Based | 一般ユーザ向け／管理者向けでストーリー群を分ける | ペルソナごとの権限差が明確なため補助的に有効 |
| Domain-Based | 「アクセス制御ドメイン」「クエリドメイン」等の業務ドメイン単位 | Feature-Basedとほぼ重複するため単独採用の必要性は低い |
| Epic-Based | 上記いずれかをエピックとして階層化し、その下に個別ストーリーを配置 | 複雑度が高いプロジェクトのため推奨 |

**提案（デフォルト）**: Feature-Based × Epic-Based のハイブリッド。requirements.md §5.0〜§5.8を9つのエピックとし、各エピック配下にFR-x.x単位（または関連するFRをまとめた単位）でストーリーを作成する。Persona-Basedの観点は各ストーリーの「As a」部分に反映する。
