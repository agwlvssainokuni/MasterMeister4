# マスタデータメンテナンスアプリケーション 要件定義書

## 1. プロジェクト概要

RDBMS に格納されたマスタデータをメンテナンスするための Web アプリケーションを開発する。
Spring Boot バックエンドと React フロントエンドによる SPA として構築する。

- **開発体制**: 単独開発者による初期開発
- **開発方針**: MVP ファーストで段階的に機能を追加する
- **優先順位**: ユーザ管理 → 対象 RDBMS セットアップ → アクセス制御 → データ表示、の順にワークフロー上流から実装する

## 3. データベースアーキテクチャ

- **内部 DB**: アプリケーションの運用データ（ユーザ、RDBMS 接続設定、取込スキーマ、グループ、権限設定、保存クエリ、クエリ実行履歴、監査ログ等）を格納する専用データベース
- **対象 RDBMS**: メンテナンス対象のマスタデータを格納するデータベース。**アプリは複数の対象 RDBMS 接続を登録・管理できる**。スキーマ取込・権限設定・監査ログは接続ごとに独立して管理される（後述 5.2）
- **コネクションプール**: 対象 RDBMS への接続パラメータは接続設定の一部として構成する。接続パスワードは内部 DB に暗号化して保存する（可逆暗号化。ユーザのログインパスワードとは異なり、接続時に再利用する必要があるため）

## 5. アプリケーション機能要件

### 5.1 ユーザ登録

2 段階（メールアドレス先行方式）の登録フローとする。

1. ユーザがメールアドレスを送信し、登録を開始する
2. 新規ユーザにのみ、有効期限付きリンクを含む登録確認メールを送信する
3. ユーザはメール内のリンクからパスワードを設定して登録を完了する
4. 登録完了したユーザは管理者ダッシュボードに一覧表示される

**管理者承認ワークフロー**
- 管理者が登録完了ユーザを承認または却下する
- 承認・却下結果をユーザにメール通知する
- 承認済みユーザのみログイン可能

**セキュリティ対策**
- メールアドレス列挙攻撃対策（ユーザの存在有無にかかわらず一貫した API レスポンスを返す）
- 登録用トークンの有効期限は設定可能（デフォルト 3 時間）
- 登録処理専用のトークン管理の仕組みを設ける

**初期管理者アカウント**
- アプリ起動時、環境変数（`mm.app.admin.bootstrap.email` / `mm.app.admin.bootstrap.password`）で指定された最初の管理者アカウントを自動作成する（通常の登録フローを経ない、初期セットアップ専用の仕組み）

**設定項目**
- トークン有効期限: `mm.app.user-registration.token-expiry`（Duration 形式、デフォルト `3h`）
- 登録確認・承認結果メール内リンクの生成に使うフロントエンドのベース URL: `mm.app.frontend.base-url`
- 登録用パス・トークンパラメータ名のカスタマイズ（**未実装**）

### 5.2 対象 RDBMS セットアップ・アクセス制御

- 管理者が管理者ダッシュボードで対象 RDBMS の接続情報を登録する。**接続は複数登録でき、それぞれ独立した ID を持つ**
- スキーマ更新操作により対象 RDBMS に接続し、テーブル/ビュー構造（物理名、コメント、型、制約）を読み取り内部 DB に取り込む。取込内容・権限設定・監査ログはすべて接続 ID に紐づけて管理する
- 管理者は取り込んだ対象 RDBMS 構造に基づき、ユーザ／グループ単位でアクセス権限を設定する

**アクセス権限モデル**

主権限（読み書きレベル）と補助権限（作成・削除の可否）を分離して管理する。

- **主権限**: スキーマ／テーブル／カラムの各階層に設定可能な 3 値。より詳細な階層（カラム＞テーブル＞スキーマ）の設定が優先する
  - `NONE`（アクセス不可）
  - `READ`（読み取りのみ）
  - `UPDATE`（既存レコードの読み取り／更新）
- **補助権限**: スキーマ／テーブル単位に設定する独立した真偽値
  - `CREATE`（レコード作成の可否）
  - `DELETE`（レコード削除の可否）
- **レコード作成可否の判定**: `CREATE` 補助権限が付与されており、かつ（主キーを持たないテーブル、または全主キー列が `UPDATE` 以上）の場合に作成可能
- **レコード削除可否の判定**: `DELETE` 補助権限が付与されており、かつ主キーを持ち、かつ全主キー列が `READ` 以上の場合に削除可能。**主キーを持たないテーブルは、`DELETE` 補助権限の有無によらず常に削除不可**
- **グループとの合成**: ユーザが所属する各グループについて同じロジックで実効権限を算出し、より許可的な方（主権限は上位、補助権限は OR）で合成する。ただし、**ユーザ自身に明示的な個別設定がある場合は、グループの合成結果より常に優先される**
- **実効権限のキャッシュ**: 算出コストを抑えるため実効権限の解決結果はキャッシュする。権限設定の変更、グループ構成の変更、スキーマ再取込のいずれかが発生した場合はキャッシュを無効化する

- アクセス権限設定は YAML 形式でエクスポート／インポート可能とする
  - **インポートは全置換方式**: 対象接続の既存の権限設定をすべて削除したうえで、YAML の内容から再構築する（マージではない）
  - YAML 内に重複エントリ（同一プリンシパル×スキーマ×テーブル×カラム、または同一プリンシパル×スキーマ×テーブル×補助権限種別）が存在する場合はインポート全体を拒否し、部分的な適用は行わない（トランザクション単位で処理する）

**グループ管理**
- 管理者はユーザグループを作成・改名・削除できる。削除時は所属関係および当該グループに設定されている権限もあわせて削除する（カスケード）
- 管理者はグループへのユーザ追加・削除を行える
- アクセス権限はユーザ個別またはグループ単位のいずれでも設定できる（合成方法は前述のとおり）

### 5.3 ユーザ認証

- ユーザ（管理者含む）はログインしてから各機能を利用する
- **JWT によるトークン認証**
  - アクセストークン: ステートレス。有効期限設定可能（デフォルト 10 分、`mm.app.jwt.access-token-expiry`）
  - リフレッシュトークン: 内部 DB に永続化（ハッシュ化して保存し、平文は保持しない）。有効期限設定可能（デフォルト 24 時間、`mm.app.jwt.refresh-token-expiry`）
  - **ローテーション方式**: リフレッシュのたびに新しいリフレッシュトークンを発行し、使用済みの旧トークンは無効化する（1 回限り使用可能）
  - **再利用検知**: 使用済みとして無効化されたトークンが再度提示された場合、トークン窃取・再送の兆候とみなし、同じセッション系列（トークンファミリ）に属する全トークンを一括で失効させる
  - ログイン (`/api/auth/login`)・リフレッシュ (`/api/auth/refresh`)・ログアウト (`/api/auth/logout`) の API を提供する

### 5.4 マスタメンテナンス機能

- **テーブル/ビュー一覧**: トップ画面にアクセス可能なテーブル/ビューを表示
- **レコード一覧**: テーブル/ビュー選択時にページング付きレコード一覧を表示
- **絞り込み**: 読み取り権限以上を持つカラムのみを対象に、フィルタ条件・ソートを直感的な UI で指定可能
- **SQL 入力**: WHERE 句・ORDER BY 句の手入力（権限に関係なく利用可能）
- **レコード編集**: レコード一覧画面上で、更新権限のあるカラムの値を編集
- **トランザクション管理**: 「反映」ボタン押下で、変更内容を単一トランザクションとしてバックエンド API に送信する。バッチ中の 1 件でも権限チェックまたは制約チェックに失敗した場合は、全体を反映せず DB への変更をゼロ件とする（オールオアナッシング）
- **レコード操作**: `CREATE` 補助権限がある場合はレコード作成、`DELETE` 補助権限がある場合はレコード削除が可能（5.2 の判定ルールに従う。主キーを持たないテーブルは削除不可）
- **統一 API**: 作成・更新・削除のすべての操作を単一トランザクションとして扱う API エンドポイントを 1 つに統合する

### 5.5 クエリビルダー機能

- **タブ構成**: SELECT / FROM / JOIN / WHERE / GROUP BY / HAVING / ORDER BY / LIMIT OFFSET
- **テーブル選択**: FROM/JOIN タブでテーブルとエイリアス（任意）を指定
- **カラム選択**: 他タブでは選択済みテーブル/エイリアスに対しアクセス可能なカラムを表示
- **集計関数**: SELECT / HAVING / ORDER BY タブで使用可能
- **SQL 生成**: UI での指定内容からスキーマ非修飾の SQL（例: `SELECT * FROM customers`）を生成する。対象スキーマを SQL に含めないことで、異なるスキーマ間で同じ SQL を使い回せるようにする（対象スキーマは実行時に指定する。5.7 参照）
- **連携**: 生成した SQL は保存または直接実行が可能
- **リバースエンジニアリング**: 他機能からの遷移時、SQL を解析してクエリビルダーの各タブに反映する

### 5.6 クエリ保存機能

- **名前付き保存**: 手入力またはクエリビルダーで作成した SQL に任意の名前を付けて保存
- **公開範囲**: 公開（Public）/ 非公開（Private）を選択可能
- **スキーマ非依存**: 保存クエリはスキーマを保持しない。クエリビルダー同様、スキーマは実行時に指定する
- **クエリ実行**: 保存済みクエリはユーザが実行可能
- **編集権限**: 保存クエリの編集は作成者のみ可能
- **非表示化**: 保存クエリを論理的に非表示（retired）にできる

### 5.7 クエリ実行機能

- **SQL 入力/編集**: 実行画面で SQL を入力・編集可能
- **実行制限**: 読み取り専用 SQL のみ実行可能
- **パラメータ対応**: NamedParameterJdbcTemplate による ":param" 形式のパラメータ化クエリをサポート
- **パラメータ入力**: パラメータを自動検出し、値を設定する UI を提供
- **実行時スキーマ指定**: SQL 実行時に対象スキーマを明示的に指定する。指定可能なスキーマは、対象接続内でユーザがアクセス権限を持つスキーマの一覧から選択させる（スキーマ名は SQL 識別子でありバインド変数化できないため、許可リスト方式で検証する）
  - スキーマの概念を持つ DB（PostgreSQL, H2）は、実行直前にスキーマ切替（`SET search_path` 等、DB 方言ごとの実装）を行う
  - カタログ（データベース）単位でスキーマが固定される DB（MySQL, MariaDB）は接続時点でスキーマが確定しているため、追加の切替操作は不要
- **ページング**: ページングの有無を設定可能
- **履歴記録**: SQL、パラメータ、対象スキーマ、結果件数、実行時間、実行回数（保存クエリの場合）、実行日時、実行者を記録
- **結果表示**: 表形式で、必要に応じてページングして表示
- **保存クエリ実行時の制限**: 保存クエリを実行する場合は編集不可

### 5.8 クエリ履歴機能

- **履歴一覧**: クエリ実行履歴をページング付きで表示
- **種別表示**: 保存クエリと直接入力クエリを区別して表示
- **絞り込み**:
  - 実行日時（範囲指定）
  - 実行者（全ユーザ／自分のみ）
  - 対象スキーマ
  - SQL テキスト検索
- **画面遷移**: 履歴からクエリ実行・クエリ保存・クエリビルダーへ遷移可能

## 6. 監査ログ要件

### 6.1 必須記録イベント

**認証イベント**
- ログイン（日時、ユーザ名。**IP アドレスの記録は未実装**）
- ログアウト（日時、ユーザ名。**セッション継続時間の記録は未実装**）
- ログイン失敗（日時、試行されたユーザ名。**IP アドレスの記録は未実装**）

**管理操作**
- アカウント登録申請
- 管理者によるアカウント承認／却下
- 対象 RDBMS 接続設定の変更
- スキーマ取り込み操作とその結果
- アクセス権限設定（テーブル/カラムレベル）の変更（グループ変更、権限設定変更、YAML エクスポート、YAML インポートを別イベント種別として区別して記録する）

**データアクセスイベント**
- 大量データ取得（閾値は設定可能、デフォルト 100 件以上）
- クエリ実行（結果件数・実行時間を含む）
- データ更新操作（挿入・更新・削除）

### 6.2 監査ログの仕様

- **形式**: 構造化ログ（JSON 推奨）
- **保存先**: 内部データベースに永続化
- **記録項目**: 日時（ISO 8601）、ユーザ ID、対象接続 ID、操作種別、対象リソース、結果ステータス。**IP アドレスの記録は未実装**
- **保持期間**: 「設定可能」を要件とするが、**保持期間設定・自動削除（パージ）機能は未実装**
- **アクセス**: 管理者のみ閲覧可能

### 6.3 セキュリティ・コンプライアンス

- 特別な要件はなし。標準的な Web アプリケーションのセキュリティ対策に準拠する
- ログ中の機微情報は基本的な保護を行う
- 監査ログへのアクセスは管理者のみに限定する

## 7. 非機能要件

### 7.1 性能・規模

- **同時利用者数**: 約 10 名を想定
- **大量データ閾値**: 監査ログ対象となるレコード件数の閾値は設定可能（デフォルト 100 件以上）
- **応答性能**: 典型的なマスタデータ操作に対して妥当な性能を確保する

### 7.2 デプロイ

**主デプロイ形態**
- Twelve-Factor App の原則に準拠する
- 自己完結型の実行可能 WAR ファイルを生成する
- すべての設定は環境変数経由で行う
- コンテナ環境に適した構造化ログを出力する

**その他のデプロイ形態**
- 実行可能 WAR から Docker コンテナ化をサポートする
- Tomcat への WAR デプロイにも対応する（将来対応）

### 7.3 メール送信

- **開発環境**: MailPit コンテナを用いてメール送受信を確認する
- **本番環境**: 環境変数経由で SMTP 設定を構成可能とする
