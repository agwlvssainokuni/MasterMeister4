# MasterMeister 要件定義書

## 0. インテント分析サマリ

- **ユーザ要求**: `aidlc-docs/inception/requirements/initial-request.md`として提供された、RDBMSマスタデータをメンテナンスするWebアプリケーション（Spring Boot + React SPA）の詳細要求文書。加えて、12問の要件確認質問と5問のクラリフィケーション質問への回答により詳細化。
- **要求タイプ**: 新規プロジェクト（New Project、Greenfield）
- **スコープ見積り**: システム全体（System-wide） — バックエンド（Spring Boot）、フロントエンド（React SPA）、開発環境（Docker Compose）、複数対象RDBMS連携を含む
- **複雑度見積り**: 複雑（Complex） — 多階層アクセス権限モデル、JWTローテーション＋再利用検知、複数RDBMS方言対応、動的SQL生成（クエリビルダー）、監査ログ、多言語対応を含む
- **要件深度**: Comprehensive（複雑度・リスクの高さに応じ、トレーサビリティ用にID付与した詳細要件とする）
- **開発体制**: 単独開発者による初期開発、MVPファーストで段階的に機能追加
- **実装優先順位（初期要求文書より）**: デザインシステム基盤（新規追加、後述）→ ユーザ管理 → 対象RDBMSセットアップ → アクセス制御 → データ表示 → その他機能

## 1. プロジェクト概要

RDBMSに格納されたマスタデータをメンテナンスするためのWebアプリケーションを開発する。Spring Bootバックエンドと React フロントエンドによるSPAとして構築する。

- **開発体制**: 単独開発者による初期開発
- **開発方針**: MVPファーストで段階的に機能を追加する
- **実装優先順位**: デザインシステム基盤 → ユーザ管理 → 対象RDBMSセットアップ → アクセス制御 → データ表示、の順に実装する（デザインシステム基盤は要件確認の結果、最優先ユニットとして追加。詳細は§5.0）

## 3. データベースアーキテクチャ

- **内部DB**: アプリケーションの運用データ（ユーザ、RDBMS接続設定、取込スキーマ、グループ、権限設定、保存クエリ、クエリ実行履歴、監査ログ等）を格納する専用データベース
- **対象RDBMS**: メンテナンス対象のマスタデータを格納するデータベース。アプリは複数の対象RDBMS接続を登録・管理できる。スキーマ取込・権限設定・監査ログは接続ごとに独立して管理される（§5.2）
- **コネクションプール**: 対象RDBMSへの接続パラメータは接続設定の一部として構成する。接続パスワードは内部DBに暗号化して保存する（可逆暗号化。ユーザのログインパスワードとは異なり、接続時に再利用する必要があるため）

## 4. プロジェクト構成

Gradleマルチモジュール構成とし、`frontend`を`backend`のサブプロジェクトとして取り込む。リリースビルド時はGradle Node Pluginによりフロントエンドをビルドし、`backend`の静的リソースとして内包した単一WARを生成する（Spring BootがSPAを配信する構成。NFR-2.2「自己完結型の実行可能WARファイルを生成する」／NFR-2.6「Tomcatへの WARデプロイにも対応する（将来対応）」に準拠。`bootWar`タスクを使用し、アプリケーションクラスに`SpringBootServletInitializer`を継承させることで、`java -jar app.war`による自己完結実行と外部Tomcatへの WARデプロイの両方に対応する）。

```
MasterMeister/
├── settings.gradle.kts   # backend, frontend をサブプロジェクトとして定義
├── backend/               # Spring Boot アプリケーション（フロントエンドのビルド成果物を内包）
├── frontend/              # React アプリケーション (Vite)
└── devenv/                # 開発環境 (Docker Compose)
```

### ビルド方針
- **バックエンド単体ビルド**: `./gradlew :backend:build`（frontendモジュールのタスクは実行されない。日常のバックエンド開発サイクルはfrontendの影響を受けない）
- **フロントエンド単体開発**: `frontend/`配下で直接`npm run dev`（HMR等、Viteの通常の開発体験をそのまま維持。Gradleを経由しない）
- **リリースビルド（単一WAR生成）**: ルートの統合タスク（例: `./gradlew bootWar`）でのみ、frontendのビルド成果物をbackendの静的リソースへコピーする処理を実行する。通常のバックエンド開発・テストではこの処理は走らない

### 開発環境（devenv/配下のDocker Compose）
- メールサーバ: MailPit（開発時のメール送受信確認用）
- 開発用データベース: MySQL / MariaDB / PostgreSQLコンテナ、H2（組み込みのためコンテナ不要）

## 5. 機能要件

### 5.0 デザインシステム基盤【新規・最優先ユニット】

要件確認の結果（Q6=C＋クラリフィケーション質問1=B、質問2=A）、以下を最優先の実装ユニットとして追加する。

- **FR-0.1**: サードパーティUIライブラリは使用せず、実際に動作するReactのUIコンポーネント一式（配色・タイポグラフィ・ボタン・フォーム部品等の基本部品）を、他機能の実装に先立って最初のユニットとして先行構築する
- **FR-0.2**: 全機能共通の基本デザイン（配色・タイポグラフィ・共通コンポーネント）をこのユニットで確定する
- **FR-0.3**: アプリ共通の画面グランドデザイン（ヘッダー・ナビゲーション・サイドバー・フッター等、全画面共通のレイアウト枠組み）をこのユニットで確定する
- **FR-0.4**: 視覚的な一貫性を早期に検証するため、以下の代表画面についてはUNIT-01でモック（デザイン確認用の画面イメージ。データ連携・業務ロジックは伴わない）を作成する。上記以外の個別画面デザインは、共通基盤確定後、各機能の実装時に順次行う（全画面を一括でモック作成することはしない、INCEPTION継続審議で一部緩和）
  - ログイン画面
  - ユーザ登録画面（メールアドレス送信・パスワード設定）
  - ユーザ管理画面（承認待ちユーザ一覧を含む）
  - マスタメンテナンス画面（一覧・編集）
  - 権限設定画面（アクセス制御のメタデータ権限設定）
- **FR-0.5**: FR-0.4のモックはデザイン確認が目的であり、各ユニットの本実装（データ連携・業務ロジック・詳細な画面仕様）は各ユニットのFunctional Design／Code Generationで別途行う。モックの見た目をそのまま踏襲するとは限らない

### 5.1 ユーザ登録

2段階（メールアドレス先行方式）の登録フローとする。

- **FR-1.1**: ユーザがメールアドレスを送信し、登録を開始する
- **FR-1.2**: 新規ユーザにのみ、有効期限付きリンクを含む登録確認メールを送信する
- **FR-1.3**: ユーザはメール内のリンクからパスワードを設定して登録を完了する
- **FR-1.4**: 登録完了したユーザは管理者向けのユーザ管理画面に一覧表示される

**管理者承認ワークフロー**
- **FR-1.5**: 管理者が登録完了ユーザを承認または却下する
- **FR-1.6**: 承認・却下結果をユーザにメール通知する
- **FR-1.7**: 承認済みユーザのみログイン可能

**セキュリティ対策**
- **FR-1.8**: メールアドレス列挙攻撃対策（ユーザの存在有無にかかわらず一貫したAPIレスポンスを返す）
- **FR-1.9**: 登録用トークンの有効期限は設定可能（デフォルト3時間）
- **FR-1.10**: 登録処理専用のトークン管理の仕組みを設ける

**パスワードポリシー（Q4=A、SECURITY-12反映）**
- **FR-1.11**: パスワードは最小文字数のみを要件とする（例: 8文字以上）。文字種混在等の複雑さ要件は課さない
- **FR-1.12**: セキュリティ拡張（SECURITY-12）に基づき、既知の漏洩パスワードリストとの照合チェックを行う
- **FR-1.13**: パスワードは適応型ハッシュアルゴリズム（例: bcrypt、Argon2）でハッシュ化して保存する

**初期管理者アカウント**
- **FR-1.14**: アプリ起動時、環境変数（`mm.app.admin.bootstrap.email` / `mm.app.admin.bootstrap.password`）で指定された最初の管理者アカウントを自動作成する（通常の登録フローを経ない、初期セットアップ専用の仕組み）

**設定項目**
- トークン有効期限: `mm.app.user-registration.token-expiry`（Duration形式、デフォルト`3h`）
- 登録確認・承認結果メール内リンクの生成に使うフロントエンドのベースURL: `mm.app.frontend.base-url`
- 登録用パス・トークンパラメータ名のカスタマイズ（**未実装**）

### 5.2 対象RDBMSセットアップ・アクセス制御

- **FR-2.1**: 管理者が管理者ダッシュボードで対象RDBMSの接続情報を登録する。接続は複数登録でき、それぞれ独立したIDを持つ
- **FR-2.2**: スキーマ更新操作により対象RDBMSに接続し、テーブル/ビュー構造（物理名、コメント、型、制約）を読み取り内部DBに取り込む。取込内容・権限設定・監査ログはすべて接続IDに紐づけて管理する
- **FR-2.3**: 管理者は取り込んだ対象RDBMS構造に基づき、ユーザ／グループ単位でアクセス権限を設定する

**アクセス権限モデル**

主権限（読み書きレベル）と補助権限（作成・削除の可否）を分離して管理する。

- **FR-2.4**: 主権限は、スキーマ／テーブル／カラムの各階層に設定可能な3値（`NONE`/`READ`/`UPDATE`）とし、より詳細な階層（カラム＞テーブル＞スキーマ）の設定が優先する
- **FR-2.5**: 補助権限は、スキーマ／テーブル単位に設定する独立した真偽値（`CREATE`/`DELETE`）とする
- **FR-2.6**: レコード作成可否は、`CREATE`補助権限が付与されており、かつ（主キーを持たないテーブル、または全主キー列が`UPDATE`以上）の場合に可能とする
- **FR-2.7**: レコード削除可否は、`DELETE`補助権限が付与されており、かつ主キーを持ち、かつ全主キー列が`READ`以上の場合に可能とする。主キーを持たないテーブルは常に削除不可
- **FR-2.8**: グループとの合成は、ユーザが所属する各グループについて同じロジックで実効権限を算出し、より許可的な方（主権限は上位、補助権限はOR）で合成する。ただしユーザ自身の明示的な個別設定はグループの合成結果より常に優先する
- **FR-2.9**: 実効権限の解決結果はキャッシュする（Caffeine）。権限設定の変更、グループ構成の変更、スキーマ再取込のいずれかが発生した場合はキャッシュを無効化する
- **PBT対象**: FR-2.4〜FR-2.9の権限判定・合成ロジックは境界値・組み合わせが多く、PBT-01（プロパティ識別）・PBT-03（不変条件）の対象として、Functional Design段階でテスト可能プロパティを識別する

**YAML入出力**
- **FR-2.10**: アクセス権限設定はYAML形式でエクスポート／インポート可能とする
- **FR-2.11**: インポートは全置換方式とする（対象接続の既存の権限設定をすべて削除したうえで、YAMLの内容から再構築する。マージではない）
- **FR-2.12**: YAML内に重複エントリが存在する場合はインポート全体を拒否し、部分的な適用は行わない（トランザクション単位で処理する）
- **PBT対象**: YAML入出力はラウンドトリップ特性（PBT-02）の対象

**グループ管理**
- **FR-2.13**: 管理者はユーザグループを作成・改名・削除できる。削除時は所属関係および当該グループに設定されている権限もあわせて削除する（カスケード）
- **FR-2.14**: 管理者はグループへのユーザ追加・削除を行える
- **FR-2.15**: アクセス権限はユーザ個別またはグループ単位のいずれでも設定できる

### 5.3 ユーザ認証

- **FR-3.1**: ユーザ（管理者含む）はログインしてから各機能を利用する
- **FR-3.2**: JWTによるトークン認証を行う
  - アクセストークン: ステートレス。有効期限設定可能（デフォルト10分、`mm.app.jwt.access-token-expiry`）
  - リフレッシュトークン: 内部DBに永続化（ハッシュ化して保存し、平文は保持しない）。有効期限設定可能（デフォルト24時間、`mm.app.jwt.refresh-token-expiry`）
- **FR-3.3**: ローテーション方式 — リフレッシュのたびに新しいリフレッシュトークンを発行し、使用済みの旧トークンは無効化する（1回限り使用可能）
- **FR-3.4**: 再利用検知 — 使用済みとして無効化されたトークンが再度提示された場合、トークン窃取・再送の兆候とみなし、同じトークンファミリに属する全トークンを一括で失効させる
- **FR-3.5**: ログイン（`/api/auth/login`）・リフレッシュ（`/api/auth/refresh`）・ログアウト（`/api/auth/logout`）のAPIを提供する
- **FR-3.6**: 同一ユーザの複数端末・複数ブラウザからの同時ログインを許可する（制限なし。Q10=A）

**ログイン試行制限（Q5=A、SECURITY-12反映）**
- **FR-3.7**: ログイン失敗が一定回数続いた場合、アカウントを一時ロックする、またはレート制限をかける

### 5.4 マスタメンテナンス機能

- **FR-4.1**: テーブル/ビュー一覧 — トップ画面にアクセス可能なテーブル/ビューを表示
- **FR-4.2**: レコード一覧 — テーブル/ビュー選択時にページング付きレコード一覧を表示
- **FR-4.3**: 絞り込み — 読み取り権限以上を持つカラムのみを対象に、フィルタ条件・ソートを直感的なUIで指定可能
- **FR-4.4**: SQL入力 — WHERE句・ORDER BY句の手入力（権限に関係なく利用可能）
- **FR-4.5**: レコード編集 — レコード一覧画面上で、更新権限のあるカラムの値を編集
- **FR-4.6**: トランザクション管理 — 「反映」ボタン押下で、変更内容を単一トランザクションとしてバックエンドAPIに送信する。バッチ中の1件でも権限チェックまたは制約チェックに失敗した場合は、全体を反映せずDBへの変更をゼロ件とする（オールオアナッシング）
- **FR-4.7**: レコード操作 — `CREATE`補助権限がある場合はレコード作成、`DELETE`補助権限がある場合はレコード削除が可能（§5.2の判定ルールに従う）
- **FR-4.8**: 統一API — 作成・更新・削除のすべての操作を単一トランザクションとして扱うAPIエンドポイントを1つに統合する

### 5.5 クエリビルダー機能

- **FR-5.1**: タブ構成 — SELECT / FROM / JOIN / WHERE / GROUP BY / HAVING / ORDER BY / LIMIT OFFSET
- **FR-5.2**: テーブル選択 — FROM/JOINタブでテーブルとエイリアス（任意）を指定
- **FR-5.3**: カラム選択 — 他タブでは選択済みテーブル/エイリアスに対しアクセス可能なカラムを表示
- **FR-5.4**: 集計関数 — SELECT / HAVING / ORDER BYタブで使用可能
- **FR-5.5**: SQL生成 — UIでの指定内容からスキーマ非修飾のSQLを生成する（対象スキーマは実行時に指定。§5.7参照）
- **FR-5.6**: 連携 — 生成したSQLは保存または直接実行が可能
- **FR-5.7**: リバースエンジニアリング — 他機能からの遷移時、SQLを解析してクエリビルダーの各タブに反映する
- **PBT対象**: SQL生成ロジック（FR-5.5）はPBT-05（オラクル比較）またはPBT-01で識別されるプロパティの対象候補

### 5.6 クエリ保存機能

- **FR-6.1**: 名前付き保存 — 手入力またはクエリビルダーで作成したSQLに任意の名前を付けて保存
- **FR-6.2**: 公開範囲 — 公開（Public）/非公開（Private）を選択可能
- **FR-6.3**: スキーマ非依存 — 保存クエリはスキーマを保持しない
- **FR-6.4**: クエリ実行 — 保存済みクエリはユーザが実行可能
- **FR-6.5**: 編集権限 — 保存クエリの編集は作成者のみ可能
- **FR-6.6**: 非表示化 — 保存クエリを論理的に非表示（retired）にできる

### 5.7 クエリ実行機能

- **FR-7.1**: SQL入力/編集 — 実行画面でSQLを入力・編集可能
- **FR-7.2**: 実行制限 — 読み取り専用SQLのみ実行可能
- **FR-7.3**: パラメータ対応 — NamedParameterJdbcTemplateによる":param"形式のパラメータ化クエリをサポート
- **FR-7.4**: パラメータ入力 — パラメータを自動検出し、値を設定するUIを提供
- **FR-7.5**: 実行時スキーマ指定 — SQL実行時に対象スキーマを明示的に指定する（許可リスト方式で検証）
  - PostgreSQL/H2: 実行直前にスキーマ切替（`SET search_path`等、DB方言ごとの実装）
  - MySQL/MariaDB: 接続時点でスキーマが確定しているため追加の切替操作は不要
- **FR-7.6**: ページング — ページングの有無を設定可能
- **FR-7.7**: 履歴記録 — SQL、パラメータ、対象スキーマ、結果件数、実行時間、実行回数（保存クエリの場合）、実行日時、実行者を記録
- **FR-7.8**: 結果表示 — 表形式で、必要に応じてページングして表示
- **FR-7.9**: 保存クエリ実行時の制限 — 保存クエリを実行する場合は編集不可

### 5.8 クエリ履歴機能

- **FR-8.1**: 履歴一覧 — クエリ実行履歴をページング付きで表示
- **FR-8.2**: 種別表示 — 保存クエリと直接入力クエリを区別して表示
- **FR-8.3**: 絞り込み — 実行日時（範囲指定）、実行者（全ユーザ／自分のみ）、対象スキーマ、SQLテキスト検索
- **FR-8.4**: 画面遷移 — 履歴からクエリ実行・クエリ保存・クエリビルダーへ遷移可能

## 6. 監査ログ要件

### 6.1 必須記録イベント

**認証イベント**
- ログイン（日時、ユーザ名。IPアドレスの記録は**未実装**）
- ログアウト（日時、ユーザ名。セッション継続時間の記録は**未実装**）
- ログイン失敗（日時、試行されたユーザ名。IPアドレスの記録は**未実装**）

**管理操作**
- アカウント登録申請
- 管理者によるアカウント承認／却下
- 対象RDBMS接続設定の変更
- スキーマ取り込み操作とその結果
- アクセス権限設定（テーブル/カラムレベル）の変更（グループ変更、権限設定変更、YAMLエクスポート、YAMLインポートを別イベント種別として区別して記録する）

**データアクセスイベント**
- 大量データ取得（閾値は設定可能、デフォルト100件以上）
- クエリ実行（結果件数・実行時間を含む）
- データ更新操作（挿入・更新・削除）

### 6.2 監査ログの仕様

- **形式**: 構造化ログ（JSON推奨）
- **保存先**: 内部データベースに永続化
- **記録項目**: 日時（ISO 8601）、ユーザID、対象接続ID、操作種別、対象リソース、結果ステータス。IPアドレスの記録は**未実装**
- **保持期間**: 「設定可能」を要件とするが、保持期間設定・自動削除（パージ）機能は**未実装**（ログは削除されず蓄積されるため、SECURITY-14の最低保持期間90日は結果的に満たされる）
- **アクセス**: 管理者のみ閲覧可能

### 6.3 セキュリティ・コンプライアンス

- 特別なコンプライアンス要件はなし。標準的なWebアプリケーションのセキュリティ対策に準拠する（詳細は§7.4）
- ログ中の機微情報は基本的な保護を行う（パスワード・トークン等の機微情報は記録しない、SECURITY-03準拠）
- 監査ログへのアクセスは管理者のみに限定する

### 7.2 デプロイ

**主デプロイ形態**
- **NFR-2.1**: Twelve-Factor Appの原則に準拠する
- **NFR-2.2**: 自己完結型の実行可能WARファイルを生成する
- **NFR-2.3**: すべての設定は環境変数経由で行う
- **NFR-2.4**: コンテナ環境に適した構造化ログを出力する

**その他のデプロイ形態**

- **NFR-3.1**: 開発環境ではMailPitコンテナを用いてメール送受信を確認する
- **NFR-3.2**: 本番環境では環境変数経由でSMTP設定を構成可能とする

### 7.4 セキュリティ（拡張ルール適用、Q1=A）

セキュリティ拡張（`.aidlc-rule-details/extensions/security/baseline/security-baseline.md`、SECURITY-01〜15）を全面適用する。以降の全ステージ（設計・コード生成）でブロッキング制約として検証する。`initial-request.md`には明記されていなかったが、本拡張ルールに基づき以下を追加要件とする。

- **NFR-4.1**: 【SECURITY-12からの文書化された例外】管理者アカウントのMFA（多要素認証）対応は行わない。SECURITY-12は「管理者アカウントへのMFA対応必須」を求めるが、本プロジェクトはユーザレビューにより意図的にこの項目を適用除外とする（理由: 同時利用者約10名規模の社内ツールであり、管理者を信頼できる小規模チームで運用するため）。他のSECURITY-12要件（パスワードポリシー、適応型ハッシュ、ブルートフォース対策、セッション管理、認証情報のハードコード禁止）は引き続き適用する
- **NFR-4.2**: 全HTML配信エンドポイントに以下のHTTPセキュリティヘッダーを設定する: `Content-Security-Policy`、`Strict-Transport-Security`、`X-Content-Type-Options`、`X-Frame-Options`、`Referrer-Policy`（SECURITY-04）
- **NFR-4.3**: 全APIエンドポイントで入力バリデーション（型・長さ・形式・サニタイズ）を実施し、SQLインジェクション対策としてパラメータ化クエリを徹底する（SECURITY-05。マスタメンテナンス・クエリビルダー・クエリ実行機能における動的SQL生成部分は特に重点対応）
- **NFR-4.4**: 依存関係は lock file で固定し、ビルド時に脆弱性スキャンを実施する（SECURITY-10）。Q12でCI/CDを今回スコープ外としたため、当面はローカルのビルド手順（例: `./gradlew dependencyCheckAnalyze`、`npm audit`）に組み込む
- **NFR-4.5**: ログイン失敗・認可失敗等のセキュリティイベントに対するアラート機構を用意する（SECURITY-14）。ただし同時利用者10名規模の社内ツールという前提に鑑み、軽量な仕組み（ログベースの検知等）で足りるものとし、本格的な監視ダッシュボードは求めない
- **NFR-4.6**: 認可チェックはアプリケーション層で徹底し（Deny by default、オブジェクトレベル認可、CORSの許可オリジン限定）、既存のアクセス権限モデル（§5.2）・JWT検証（§5.3）と統合する（SECURITY-08）
- **NFR-4.7**: リフレッシュトークンのクライアント保持方式（httpOnly Cookie等）はXSS対策の観点から、NFR Design段階で確定する（SECURITY-12）。【UNIT-02 NFR Designで確定】アクセストークン・リフレッシュトークンとも、レスポンスボディで返却しクライアント側`sessionStorage`に保管する方式とした。Cookieを用いないためCSRF対策は不要となる一方、XSS対策（CSP等、NFR-4.2/SECURITY-04）が主たる防御層となる
- **NFR-4.8**: 【SECURITY-01（保存時暗号化）からの文書化された例外】内部DB（H2 Database）のアプリケーション層での暗号化（DBファイル自体の暗号化）は行わない。パスワード・トークン等の機微情報は個別にハッシュ化して保存済みであり（SECURITY-12準拠）、平文で保存されるのは氏名・メールアドレス等のPIIに限られる。同時利用者約10名規模の社内ツールという前提のもと、DBファイル自体の暗号化はホスト・ディスクレベルの暗号化（デプロイ環境が提供）に委ねる運用要件とし、アプリ層での追加実装（鍵管理の複雑化を伴う）は行わない。通信経路の暗号化（SECURITY-01のもう一方の要件）は、内部DBがファイルベースの組み込みH2でありネットワーク経由のアクセスを行わないため、本項の対象外とする
- 上記以外のSECURITY-01〜15の各ルールは、該当する設計・実装ステージで個別に適用可否を判定し、適用対象外の場合はN/Aとして記録する

### 7.5 プロパティベーステスト（拡張ルール適用、Q3=A）

PBT拡張（`.aidlc-rule-details/extensions/testing/property-based/property-based-testing.md`、PBT-01〜10）を全面適用する。

- **NFR-5.1**: PBT対象候補として、権限判定・合成ロジック（§5.2 FR-2.4〜FR-2.9）、YAML入出力（FR-2.10〜FR-2.12）、SQL生成ロジック（FR-5.5）を識別済み。Functional Design段階でPBT-01に基づき正式なプロパティ一覧を確定する
- **NFR-5.2**: PBTフレームワークはNFR Requirements段階で最終確定する（候補: バックエンドはjqwik）
- **NFR-5.3**: PBTは例示ベーステスト（JUnit5+Mockito、Vitest+RTL）を代替せず補完するものとする（PBT-10）

### 7.6 レジリエンシー（拡張スキップ、Q2=B）

レジリエンシーベースライン拡張は適用しない。同時利用者数10名規模の社内ツール・単独開発・MVPファーストという前提に基づく判断。将来的にビジネスクリティカルな用途に発展する場合は再検討する。

### 7.7 UI/UXデザインシステム（Q6=C＋クラリフィケーション）

- **NFR-6.1**: サードパーティUIライブラリは使用しない
- **NFR-6.2**: デザインシステム基盤（§5.0）を最優先ユニットとして先行構築する

### 7.8 多言語対応（Q7=C＋クラリフィケーション）

- **NFR-7.1**: 初期リリースから日本語＋英語の2言語に対応する
- **NFR-7.2**: 適用範囲は、画面文言・フォームラベル・エラーメッセージに加え、確認・通知メール文面、監査ログの表示など、ユーザに見えるテキストすべてとする
- **NFR-7.3**: 各機能の実装と同時に多言語対応を組み込む（後からまとめて多言語化する手戻りは行わない）。バックエンド・フロントエンドともにi18n基盤（文言の外部化）は最初の実装ユニットから導入する

### 7.9 対応ブラウザ・デバイス（Q9=B）

- **NFR-8.1**: デスクトップ利用を中心とし、タブレット等でも大きく崩れない程度のレスポンシブ対応を行う（モバイル本格対応は不要）

### 7.10 テスト戦略（Q11=A）

- **NFR-9.1**: バックエンドはJUnit5 + Mockito、フロントエンドはVitest + React Testing Libraryを標準の組み合わせとする
- **NFR-9.2**: PBT対象範囲は§7.5参照

### 7.11 CI/CD（Q12=B、追加指示あり）

- **NFR-10.1**: 今回のスコープにCI/CDパイプラインの構築は含めない。ローカルでのビルド・テスト手順の整備までとし、CI/CDは開発の最終段階で対応する（Build and Testステージでローカル実行手順を整備する）
- **NFR-10.2**: CI/CDはGitHub Actionsで構成する。開発が一通り進んだ最終段階のユニットとして実施する
- **NFR-10.3**: リリース手順として、タグpushをトリガーにGitHub Releasesを作成する仕組みを構築する

### 7.12 API仕様書（Q8=A）

- **NFR-11.1**: OpenAPI仕様を自動生成し、Swagger UI等で閲覧可能にする

## 9. 要件サマリ

**主要な機能要件**
- デザインシステム基盤を最優先ユニットとして先行構築（サードパーティUIライブラリ不使用）
- 2段階（メール先行）ユーザ登録＋管理者承認ワークフロー
- 複数対象RDBMS（MySQL/MariaDB/PostgreSQL/H2）の接続管理とスキーマ取込
- 主権限（NONE/READ/UPDATE）＋補助権限（CREATE/DELETE）による多階層アクセス権限モデル、ユーザ・グループ合成、YAML入出力
- JWTベース認証（アクセス＋リフレッシュトークン、ローテーション、再利用検知）、同時多重ログイン許可
- マスタメンテナンス（一覧・絞込・編集・オールオアナッシングの一括反映）
- クエリビルダー（タブUI・SQL生成・リバースエンジニアリング）、クエリ保存（公開/非公開）、クエリ実行（読み取り専用・パラメータ化）、クエリ履歴
- 認証・管理操作・データアクセスの監査ログ記録
- 初期リリースから日本語＋英語の多言語対応（全範囲、各機能実装と同時に組み込み）

**主要な非機能要件**
- 同時利用者約10名規模、Twelve-Factor App準拠・自己完結型WAR・環境変数設定
- セキュリティ拡張ルール（SECURITY-01〜15）全面適用— MFA（管理者）、HTTPセキュリティヘッダー、入力バリデーション、依存関係脆弱性スキャン、認証失敗アラート等を追加要件として明記
- プロパティベーステスト拡張（PBT-01〜10）全面適用 — 権限判定ロジック・YAML入出力・SQL生成ロジックを重点対象として識別
- レジリエンシーベースラインは適用しない（規模・開発体制を考慮）
- デスクトップ中心・軽量レスポンシブ対応、API仕様書（OpenAPI）は今回スコープに含める
- CI/CDは開発の最終段階でGitHub Actionsにより構成し、タグpushをトリガーにGitHub Releasesを作成する
