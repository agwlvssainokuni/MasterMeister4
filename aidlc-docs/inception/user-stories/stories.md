# ユーザストーリー

`story-generation-plan.md`の回答（Q1=A 2ペルソナ、Q2=B 機能単位の粒度、Q3=A Feature-Based×Epic-Based、Q4=A Given-When-Then、Q5=A PBT対象を明記）に基づき作成。

各ストーリーはrequirements.mdのFR-x.xへのトレーサビリティを「対応要件」に明記する。優先順位はrequirements.md §1の実装優先順位（デザインシステム基盤→ユーザ管理→対象RDBMSセットアップ→アクセス制御→データ表示→その他機能）に準拠し、P0（最優先）〜P6で表す。監査ログ（§6、FR-IDなし）は横断的要件のため、記録は関連ストーリーの受け入れ基準に含め、閲覧機能のみ独立エピックとした。CI/CD（NFR-10.x）はGitHub Actions構成という開発タスクでありユーザストーリーの対象外のため、本ドキュメントには含めない（Workflow Planningで最終ユニットとして計画する）。

**INVEST準拠**: 各ストーリーはIndependent・Negotiable・Valuable・Estimable・Small・Testableであることを意図して作成。

---

## Epic 0: デザインシステム基盤（優先度 P0）

### STORY-0.1: 共通デザイン基盤の構築
**As a** 一般ユーザ・管理者として
**I want** 一貫した配色・タイポグラフィ・基本UIコンポーネント（ボタン、フォーム部品等）で構成された画面を使いたい
**So that** どの機能を使っても迷わず操作でき、学習コストが低い

- **対応要件**: FR-0.1, FR-0.2, FR-0.3
- **PBT対象**: いいえ（UIコンポーネントのため対象外）
- **受け入れ基準**:
  - Given 開発チームが最初の実装ユニットに着手する, When 共通デザイン基盤（配色・タイポグラフィ・ボタン・フォーム等）を構築する, Then サードパーティUIライブラリを使わずに独自コンポーネントとして提供される
  - Given 共通デザイン基盤が完成している, When 以降の機能画面を実装する, Then その基盤コンポーネントを再利用できる
  - Given 共通デザイン基盤を構築する, When ヘッダー・ナビゲーション・サイドバー・フッター等の画面グランドデザイン（共通レイアウト枠組み）を確定する, Then 以降の全画面がその共通レイアウト枠組みに従って構成される

### STORY-0.2: 個別画面デザインの段階的整備
**As a** 一般ユーザ・管理者として
**I want** 各機能画面が実装される都度、その機能に適したレイアウトで提供されてほしい
**So that** 全画面を待たずに実装優先順位どおり機能を使い始められる

- **対応要件**: FR-0.4
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given 共通デザイン基盤が確定している, When 個別機能（ユーザ登録画面等）を実装する, Then その機能の実装タイミングで画面デザインも合わせて整備される（代表画面モック以外の全画面を一括先行デザインすることは行わない）

### STORY-0.3: 代表画面モックによる早期デザイン確認
**As a** 開発チーム（および画面デザインをレビューする関係者）として
**I want** 代表的な画面（ログイン、ユーザ登録、管理者ダッシュボード、マスタメンテナンス、権限設定）のモックを早期に確認したい
**So that** 個別機能を実装する前に画面デザインの一貫性・妥当性を検証できる

- **対応要件**: FR-0.4, FR-0.5
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given 共通デザイン基盤とグランドデザインが確定している, When 代表5画面（ログイン、ユーザ登録、管理者ダッシュボード、マスタメンテナンス、権限設定）のモックを作成する, Then データ連携・業務ロジックを伴わない、デザイン確認用の画面イメージとして提供される
  - Given 代表画面のモックが作成されている, When 各ユニットで該当機能を実装する, Then モックのデザインを参考にしつつ、実際のデータ連携・業務ロジックを踏まえて改めて画面を実装する（モックの見た目をそのまま踏襲するとは限らない）

## Epic 1: ユーザ登録（優先度 P1）

### STORY-1.1: メールアドレスによる登録開始
**As a** 一般ユーザ（登録希望者）として
**I want** メールアドレスを入力するだけでアカウント登録を開始したい
**So that** 最小限の入力で登録手続きを始められる

- **対応要件**: FR-1.1, FR-1.2, FR-1.8, FR-1.9, FR-1.10
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given 未登録のメールアドレスを持つ利用希望者がいる, When メールアドレスを送信する, Then 有効期限付きの登録確認メールが送信される
  - Given 既に登録済みのメールアドレスが送信される, When 登録リクエストを処理する, Then 未登録の場合と同一のAPIレスポンスを返す（メールアドレス列挙攻撃対策）
  - Given 登録用トークンが発行されている, When 設定された有効期限（デフォルト3時間）を過ぎる, Then トークンは無効となる

### STORY-1.2: パスワード設定による登録完了
**As a** 一般ユーザ（登録希望者）として
**I want** メール内リンクからパスワードを設定して登録を完了したい
**So that** 自分だけが知るパスワードでアカウントを保護できる

- **対応要件**: FR-1.3, FR-1.11, FR-1.12, FR-1.13
- **PBT対象**: いいえ（パスワード強度チェックは単純な条件判定のため対象外と判断。ただし境界値の例示テストは必須）
- **受け入れ基準**:
  - Given 有効な登録リンクを開いている, When 最小文字数（例: 8文字以上）を満たすパスワードを設定する, Then 登録が完了する
  - Given 既知の漏洩パスワードリストに一致するパスワードを入力する, When 登録を試みる, Then エラーとなり別のパスワードを求められる
  - Given パスワードが保存される, When 内部DBに永続化する, Then 適応型ハッシュアルゴリズムでハッシュ化されて保存される（平文は保持しない）

### STORY-1.3: 管理者によるユーザ承認・却下
**As a** 管理者として
**I want** 登録完了したユーザの一覧を確認し、承認または却下したい
**So that** 意図しないユーザにシステムを使わせないようにできる

- **対応要件**: FR-1.4, FR-1.5, FR-1.6, FR-1.7
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given 登録が完了したユーザが存在する, When 管理者ダッシュボードを開く, Then 承認待ちユーザが一覧表示される
  - Given 管理者がユーザを承認する, When 承認処理を実行する, Then ユーザにメール通知が送られ、以後ログイン可能になる
  - Given 管理者がユーザを却下する, When 却下処理を実行する, Then ユーザにメール通知が送られ、ログインはできないままとなる
  - Given 承認・却下操作が行われる, When 処理が完了する, Then 監査ログに記録される（§6.1管理操作）

### STORY-1.4: 初期管理者アカウントのブートストラップ
**As a** システム運用者（初回セットアップ担当）として
**I want** アプリ初回起動時に環境変数で指定した管理者アカウントを自動作成したい
**So that** 通常の登録・承認フローを経ずに最初の管理者でログインできる

- **対応要件**: FR-1.14
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given `mm.app.admin.bootstrap.email`と`mm.app.admin.bootstrap.password`が環境変数として設定されている, When アプリが初回起動する, Then 承認済みの管理者アカウントが自動作成される

  - Given 複数の接続がすでに登録されている, When さらに接続を追加する, Then 既存の接続に影響なく独立して管理される

  - Given スキーマ取込を実行する, When 処理が完了する, Then 取込結果が接続IDに紐づけて記録され、監査ログにも記録される（§6.1管理操作）

## Epic 2続き: アクセス制御（優先度 P3）

### STORY-2.3: アクセス権限の設定（主権限・補助権限）
**As a** 管理者として
**I want** 取り込んだスキーマ/テーブル/カラムに対してユーザ・グループ単位でアクセス権限を設定したい
**So that** 各ユーザに必要最小限の権限だけを与えられる

- **対応要件**: FR-2.3, FR-2.4, FR-2.5
- **PBT対象**: いいえ（設定UI自体は対象外。判定ロジックはSTORY-2.4参照）
- **受け入れ基準**:
  - Given スキーマ取込が完了している, When 管理者がユーザまたはグループにスキーマ／テーブル／カラム単位の主権限（NONE/READ/UPDATE）を設定する, Then より詳細な階層（カラム＞テーブル＞スキーマ）の設定が優先される
  - Given 管理者が補助権限（CREATE/DELETE）をテーブル単位で設定する, When 設定を保存する, Then 主権限とは独立した真偽値として保存される
  - Given 権限設定が変更される, When 変更が保存される, Then 監査ログに記録される（グループ変更・権限設定変更を区別、§6.1）

### STORY-2.4: 実効権限の判定ロジック
**As a** 一般ユーザとして
**I want** 自分の個別設定と所属グループの設定が正しく合成された実効権限で操作したい
**So that** 意図した範囲でのみレコードの参照・編集・作成・削除ができる

- **対応要件**: FR-2.6, FR-2.7, FR-2.8, FR-2.9
- **PBT対象**: **はい** — 主権限/補助権限の合成、作成・削除可否判定は境界値・組み合わせが多く、property-based testingによる網羅的検証が有効（PBT-01, PBT-03）
- **受け入れ基準**:
  - Given `CREATE`補助権限が付与されている, When 対象テーブルが主キーを持たない、または全主キー列が`UPDATE`以上である, Then レコード作成が可能と判定される
  - Given `DELETE`補助権限が付与されている, When 対象テーブルが主キーを持ち、かつ全主キー列が`READ`以上である, Then レコード削除が可能と判定される。主キーを持たないテーブルは常に削除不可と判定される
  - Given ユーザが複数グループに所属している, When 実効権限を算出する, Then 各グループの権限がより許可的な方（主権限は上位、補助権限はOR）で合成される
  - Given ユーザ自身に個別設定がある, When グループの合成結果と比較する, Then ユーザ個別設定が常に優先される
  - Given 権限設定・グループ構成・スキーマ再取込のいずれかが変更される, When 変更が確定する, Then 実効権限のキャッシュが無効化される

### STORY-2.5: アクセス権限のYAMLエクスポート／インポート
**As a** 管理者として
**I want** アクセス権限設定をYAML形式でエクスポート・インポートしたい
**So that** 権限設定を一括でバックアップ・移行・レビューできる

- **対応要件**: FR-2.10, FR-2.11, FR-2.12
- **PBT対象**: **はい** — エクスポート→インポートのラウンドトリップ特性（PBT-02）の検証対象
- **受け入れ基準**:
  - Given 現在のアクセス権限設定がある, When YAMLエクスポートを実行する, Then 現在の設定内容を過不足なく表すYAMLが出力される
  - Given YAMLをインポートする, When インポートを実行する, Then 対象接続の既存権限設定がすべて削除された上でYAMLの内容から再構築される（全置換方式、マージではない）
  - Given YAML内に重複エントリ（同一プリンシパル×スキーマ×テーブル×カラム、または同一プリンシパル×スキーマ×テーブル×補助権限種別）が存在する, When インポートを実行する, Then インポート全体が拒否され、部分適用は行われない
  - Given YAMLエクスポート・インポートが実行される, When 処理が完了する, Then それぞれ別イベント種別として監査ログに記録される（§6.1）

### STORY-2.6: グループ管理
**As a** 管理者として
**I want** ユーザグループを作成・改名・削除し、ユーザの所属を管理したい
**So that** 権限を個別設定ではなくグループ単位でまとめて管理できる

- **対応要件**: FR-2.13, FR-2.14, FR-2.15
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given グループが存在する, When 管理者がグループを削除する, Then 所属関係および当該グループに設定された権限もカスケードで削除される
  - Given グループが存在する, When 管理者がユーザを追加・削除する, Then 対象ユーザの実効権限算出（STORY-2.4）に反映される
  - Given グループ変更が行われる, When 変更が保存される, Then 監査ログに記録される（§6.1）

## Epic 3: ユーザ認証（優先度 P1、Epic 1と並行して実装）

### STORY-3.1: ログイン・ログアウト・トークンリフレッシュ
**As a** 一般ユーザ・管理者として
**I want** ログイン・ログアウトでき、複数端末からの同時利用もできるようにしたい
**So that** 会社PC・自宅PC等どこからでも利用できる

- **対応要件**: FR-3.1, FR-3.2, FR-3.5, FR-3.6
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given 承認済みユーザが正しい認証情報を入力する, When `/api/auth/login`を呼び出す, Then アクセストークン（デフォルト10分）とリフレッシュトークン（デフォルト24時間）が発行される
  - Given 有効なリフレッシュトークンを持つ, When `/api/auth/refresh`を呼び出す, Then 新しいアクセストークンが発行される
  - Given ログイン中のユーザが`/api/auth/logout`を呼び出す, When ログアウト処理が完了する, Then 該当セッションのトークンが無効化される
  - Given 同一ユーザが複数端末から同時にログインする, When それぞれ認証する, Then いずれの端末のセッションも制限なく有効である
  - Given ログイン・ログアウト・ログイン失敗が発生する, When 各イベントが処理される, Then 監査ログに記録される（§6.1認証イベント）

### STORY-3.2: リフレッシュトークンのローテーションと再利用検知
**As a** システムのセキュリティ担当（管理者視点）として
**I want** リフレッシュトークンが盗用された場合に自動検知・失効させたい
**So that** トークン漏洩による不正アクセスの被害を最小化できる

- **対応要件**: FR-3.3, FR-3.4
- **PBT対象**: いいえ（状態遷移が明確で例示ベーステストで十分検証可能なため）
- **受け入れ基準**:
  - Given リフレッシュトークンが使用される, When 新しいトークンが発行される, Then 使用済みの旧トークンは無効化される（1回限り使用可能）
  - Given 無効化された旧トークンが再度リフレッシュに使用される, When 再利用が検知される, Then 同じトークンファミリに属する全トークンが一括で失効する

### STORY-3.3: ログイン試行制限
**As a** システムのセキュリティ担当（管理者視点）として
**I want** ログイン失敗が続いたアカウントを一時的にロックまたは制限したい
**So that** パスワードの総当たり攻撃からアカウントを保護できる

- **対応要件**: FR-3.7
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given ログイン失敗が一定回数続く, When 閾値に達する, Then アカウントが一時的にロックされる、またはレート制限がかかる

## Epic 4: マスタメンテナンス（優先度 P4、データ表示）

### STORY-4.1: テーブル/ビュー一覧とレコード一覧の表示
**As a** 一般ユーザとして
**I want** アクセス可能なテーブル/ビューの一覧と、選択したテーブルのレコード一覧を見たい
**So that** 自分が担当するマスタデータをすぐに確認できる

- **対応要件**: FR-4.1, FR-4.2
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given ユーザが実効権限を持つテーブル/ビューが存在する, When トップ画面を開く, Then アクセス可能なテーブル/ビューのみが一覧表示される
  - Given テーブルを選択する, When レコード一覧画面に遷移する, Then ページング付きでレコードが表示される

### STORY-4.2: 絞り込み・SQL入力によるレコード検索
**As a** 一般ユーザとして
**I want** フィルタUIまたは直接SQL（WHERE/ORDER BY）でレコードを絞り込みたい
**So that** 目的のレコードを素早く見つけられる

- **対応要件**: FR-4.3, FR-4.4
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given ユーザが読み取り権限以上を持つカラムがある, When フィルタ条件・ソートを指定する, Then そのカラムのみを対象とした絞り込みができる
  - Given ユーザがWHERE句・ORDER BY句を手入力する, When 検索を実行する, Then 権限に関係なく実行できる

### STORY-4.3: レコード編集と一括反映
**As a** 一般ユーザとして
**I want** 複数レコードの変更内容をまとめて1回で反映したい
**So that** 中途半端な状態でデータが更新されるリスクを避けられる

- **対応要件**: FR-4.5, FR-4.6, FR-4.8
- **PBT対象**: いいえ（オールオアナッシングの動作は例示ベーステストで検証。トランザクション境界の正しさが主眼）
- **受け入れ基準**:
  - Given 更新権限のあるカラムがある, When レコード一覧画面で値を編集する, Then 編集内容が画面上に保持される
  - Given 複数件の変更が保留されている, When 「反映」ボタンを押下する, Then 全変更が単一トランザクションとして統一APIに送信される
  - Given バッチ中の1件でも権限チェックまたは制約チェックに失敗する, When 反映処理を実行する, Then DBへの変更はゼロ件となる（オールオアナッシング）
  - Given データ更新操作が成功する, When 処理が完了する, Then 監査ログに記録される（§6.1データアクセスイベント）

### STORY-4.4: レコード作成・削除
**As a** 一般ユーザとして
**I want** 権限がある場合にレコードを作成・削除したい

## Epic 5: クエリビルダー（優先度 P5、その他機能）

### STORY-5.1: タブUIによるSQL組み立て

**I want** SELECT/FROM/JOIN/WHERE/GROUP BY/HAVING/ORDER BY/LIMIT OFFSETをタブで指定したい

- **対応要件**: FR-5.1, FR-5.2, FR-5.3, FR-5.4
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given FROM/JOINタブでテーブルとエイリアスを指定する, When 他タブに移動する, Then 選択済みテーブル/エイリアスに対しアクセス可能なカラムのみが表示される
  - Given SELECT/HAVING/ORDER BYタブを操作する, When 集計関数を指定する, Then 集計関数を含むクエリ条件を組み立てられる

### STORY-5.2: SQL生成と実行・保存への連携
**As a** 一般ユーザとして
**I want** タブUIでの指定内容から実行可能なSQLを生成したい
**So that** 生成したSQLをそのまま実行・保存できる

- **対応要件**: FR-5.5, FR-5.6
- **PBT対象**: **はい** — UI指定内容とSQL生成結果の対応関係はオラクル比較（PBT-05）またはラウンドトリップ的性質の検証対象（例: 生成SQLをリバースエンジニアリングした結果が元のUI状態と一致するか）
- **受け入れ基準**:
  - Given タブUIでクエリ条件を指定する, When SQL生成を実行する, Then スキーマ非修飾のSQL（例: `SELECT * FROM customers`）が生成される
  - Given SQLが生成される, When ユーザが連携操作を行う, Then クエリ保存またはクエリ実行に直接遷移できる

### STORY-5.3: 既存SQLからのリバースエンジニアリング

**I want** 既存のSQL（保存クエリ・履歴等）をクエリビルダーの各タブに反映したい
**So that** ゼロから組み立てずに既存クエリを微調整できる

- **対応要件**: FR-5.7
- **PBT対象**: いいえ（STORY-5.2のSQL生成PBTでラウンドトリップとして間接的にカバー）
- **受け入れ基準**:
  - Given 他機能（クエリ保存・クエリ履歴）からクエリビルダーへ遷移する, When SQLが解析される, Then 各タブ（SELECT/FROM/JOIN等）に解析結果が反映される

## Epic 6: クエリ保存（優先度 P5）

### STORY-6.1: クエリの名前付き保存と公開範囲設定
**As a** 一般ユーザとして
**I want** 作成したSQLに名前を付けて保存し、公開・非公開を選べるようにしたい
**So that** よく使うクエリを再利用したり、チームで共有したりできる

- **対応要件**: FR-6.1, FR-6.2, FR-6.3
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given 手入力またはクエリビルダーで作成したSQLがある, When 名前を付けて保存する, Then 公開（Public）/非公開（Private）を選択して保存できる
  - Given 保存クエリが存在する, When 別ユーザがそのクエリにアクセスする, Then 公開範囲設定（Public/Private）に従って参照可否が決まる
  - Given 保存クエリを実行・参照する, When スキーマを指定する, Then クエリ自体はスキーマを保持せず、実行時に指定したスキーマが適用される

### STORY-6.2: 保存クエリの実行・編集・非表示化
**As a** 一般ユーザとして
**I want** 保存したクエリを実行したり、作成者として編集・非表示化したりしたい
**So that** 保存クエリのライフサイクルを自分で管理できる

- **対応要件**: FR-6.4, FR-6.5, FR-6.6
- **PBT対象**: いいえ
- **受け入れ基準**:
  - Given 保存クエリが存在する, When ユーザが実行する, Then クエリ実行機能（Epic7）に連携される
  - Given 保存クエリの作成者ではないユーザが編集を試みる, When 編集操作を行う, Then 編集は拒否される
  - Given 保存クエリの作成者が非表示化を行う, When 処理が完了する, Then 論理的に非表示（retired）となり、一覧に表示されなくなる

**As a** 一般ユーザとして
**I want** 実行対象のスキーマを明示的に指定したい

  - Given 結果が返る, When ページングが有効に設定されている, Then 表形式でページングして表示される
  - Given 保存クエリを実行する, When 実行画面を開く, Then SQLは編集不可として表示される
