# UNIT-04 アクセス制御 - Functional Design 計画

## ユニットスコープ（前提確認）

対応要件: FR-2.3〜FR-2.15（アクセス権限モデル、実効権限判定、YAML入出力、グループ管理）
対応コンポーネント: COMP-10 AccessControlService, COMP-11 EffectivePermissionResolver, COMP-12 PermissionYamlService
対応ストーリー: STORY-2.3, STORY-2.4, STORY-2.5, STORY-2.6

既存の決定事項（requirements.md §5.2/§7.5）:
- 主権限は3値（`NONE`/`READ`/`UPDATE`）、階層はカラム＞テーブル＞スキーマ（FR-2.4）
- 補助権限は`CREATE`/`DELETE`の独立した真偽値、スキーマ／テーブル単位（FR-2.5）
- レコード作成可否・削除可否の判定式（FR-2.6/2.7）、グループ合成方式（FR-2.8）は確定済み
- 実効権限の解決結果はCaffeineでキャッシュし、権限変更/グループ変更/スキーマ再取込で無効化する（FR-2.9）
- YAML入出力は全置換方式、重複エントリ検出時は全体拒否（FR-2.10〜2.12）
- グループ削除はカスケード（所属関係・権限設定も削除、FR-2.13）
- PBT対象: FR-2.4〜2.9（権限判定・合成、PBT-01/03）、YAML入出力のラウンドトリップ（FR-2.10〜2.12、PBT-02）

### Question 2（Domain Model・重要な設計上の注意点）
権限設定の対象（スキーマ／テーブル／カラム）は、UNIT-03で取り込んだ`SchemaSnapshot`/`SchemaTable`/`SchemaColumn`エンティティへの外部キー参照としますか、それとも名前（文字列）で独立して保持しますか？

**注意**: UNIT-03の`refreshSchema()`は既存`SchemaSnapshot`を削除してから全置換で作り直す方式（BR-RDBMS-08）です。もし外部キー参照（FK）にすると、スキーマを再取込するたびに紐づく権限設定がカスケード削除で全消失してしまいます。

A) 名前（schemaName/tableName/columnNameの文字列）で独立して保持する。スキーマ再取込があっても権限設定は保持され、対象が削除されていれば単に「参照先なし」として扱う（実効権限判定時に該当テーブル自体が存在しなければ判定不能＝実質NONE）

B) `SchemaTable`/`SchemaColumn`への外部キー参照とする（再取込のたびに権限が失われる制約を許容する）

C) Other（[Answer]: の後に内容を記述）

[Answer]: A

### Question 5（Business Rules・重要な曖昧点）
FR-2.8「ユーザ自身の明示的な個別設定はグループの合成結果より常に優先する」について、優先関係は階層レベル単位で判定しますか、それとも「個別設定が1件でもあれば丸ごとグループ設定を無視する」という全有無判定ですか？

A) レベル単位で判定する（例: あるテーブルにユーザ個別のカラム権限があっても、別テーブルではユーザ個別設定がなければ通常どおりグループ合成結果が適用される。カラム＞テーブル＞スキーマの階層優先順位は、ユーザ個別設定・グループ合成結果それぞれの中で先に解決してから、両者を比較する）

B) プリンシパル全体で判定する（そのユーザに個別設定が1件でも存在すれば、全リソースに対してグループ設定は一切参照しない）

C) Other（[Answer]: の後に内容を記述）

[Answer]: A（誤解のないよう例をたくさんあげて確認して欲しい）
