# UNIT-04 アクセス制御 - Business Logic Model

`unit-04-functional-design-plan.md`の回答（Q1=A, Q2=A, Q3=A, Q4=A, Q5=A, Q6=A, Q7=A, Q8=A, Q9=A, Q10=A）に基づく。技術非依存を原則とし、Caffeineキャッシュの具体的な設定値等の技術選定はNFR Requirements/NFR Designで扱う。

対応コンポーネント: COMP-10（AccessControlService）, COMP-11（EffectivePermissionResolver）, COMP-12（PermissionYamlService）

**前提（UNIT-03との関係）**: UNIT-03は当初「1接続=1スキーマ」という前提だったが、`initial-request.md` §5.7・FR-7.5（実行時スキーマ指定、対象接続内でユーザがアクセス権限を持つスキーマの一覧から選択）の本来の想定に合わせ、1接続内に複数スキーマが存在しうる前提に訂正した（PostgreSQL/H2はシステムスキーマを除く全スキーマを自動検出。MySQL/MariaDBは「データベース」がスキーマに相当する単位のため常に1件）。そのためFR-2.4の3階層（スキーマ／テーブル／カラム）はそのまま実際のスキーマ単位で表現する。

---

## 1. アクセス権限の設定（FR-2.3〜FR-2.5）

1. 管理者が、ある接続（`connectionId`）に対して、あるプリンシパル（ユーザまたはグループ）・あるリソース（スキーマ／テーブル／カラムの階層。BR-ACCESS-01）に、主権限（`NONE`/`READ`/`UPDATE`）と補助権限（`CREATE`/`DELETE`、スキーマ・テーブル単位のみ設定可能）を設定する（`AccessControlService.setPermission()`）
2. 設定は当該プリンシパル・当該リソースの既存設定を上書きする（同一キーへの再設定はupsert）
3. 「未設定」に戻す操作（一度設定した権限を解除し、グループ合成結果や既定値に委ねる状態に戻す）も可能とする。未設定に戻すと、当該プリンシパル・当該リソースの`AccessPermission`エントリ自体を削除する（BR-ACCESS-01）
4. 設定変更のたびに`EffectivePermissionResolver.invalidateCache()`を呼び出し、実効権限キャッシュを無効化する（FR-2.9、§3手順5）
5. 設定変更はAuditEventPublisher経由で`PERMISSION_CHANGED`イベントとして記録する（BR-ACCESS-12）

## 2. 実効権限の判定ロジック（FR-2.6〜FR-2.9、PBT-01/03対象）

`EffectivePermissionResolver.resolvePrimary(userId, connectionId, resource)`の判定手順。

### 2.1 階層優先順位（同一プリンシパル内、FR-2.4）
あるプリンシパル（ユーザ自身、または特定の1グループ）が持つ設定の中では、カラム＞テーブル＞スキーマの順で、最も詳細な階層に設定があればそれを採用する。

### 2.2 個別設定とグループ合成の優先判定（FR-2.8、Q5=A、BR-ACCESS-04）
1. まず、ユーザ自身の設定について§2.1の階層優先順位で解決する
2. ユーザ自身の設定が対象リソースについて**1件でも存在すれば**（スキーマ／テーブル／カラムいずれかの階層で）、その解決結果を実効権限として確定する（グループの設定は参照しない）
3. ユーザ自身の設定が対象リソースについて**一切存在しなければ**、ユーザが所属する各グループについて同様に§2.1の階層優先順位で解決し、グループ間はより許可的な方（主権限は`NONE < READ < UPDATE`の順で上位、補助権限は論理和）で合成した結果を実効権限とする
4. どの設定も存在しない場合はデフォルト値`NONE`（主権限）／`false`（補助権限）とする（Q4=A、BR-ACCESS-03）

具体例による確認はBR-ACCESS-04参照。

### 2.3 レコード作成可否（FR-2.6、`canCreate()`）
`CREATE`補助権限（テーブル階層またはスキーマ階層で解決した値）が`true`であり、かつ（対象テーブルが主キーを持たない、または全主キー列の実効主権限が`UPDATE`）の場合に作成可能と判定する（BR-ACCESS-06）。

### 2.4 レコード削除可否（FR-2.7、`canDelete()`）
`DELETE`補助権限が`true`であり、かつ対象テーブルが主キーを持ち、かつ全主キー列の実効主権限が`READ`以上（`READ`または`UPDATE`）の場合に削除可能と判定する。主キーを持たないテーブルは常に削除不可（BR-ACCESS-07）。

### 2.5 実効権限キャッシュ（FR-2.9）
`resolvePrimary()`/`canCreate()`/`canDelete()`の結果はキャッシュする（Caffeine、具体的なサイズ・TTLはNFR Requirements/NFR Designで確定）。以下のいずれかが発生した場合、`invalidateCache()`を呼び出しキャッシュを無効化する（BR-ACCESS-08）。
- 権限設定の変更（§1）
- グループ構成の変更（グループ作成/改名/削除、所属追加/削除、§4）
- スキーマ再取込（UNIT-03の`SCHEMA_IMPORTED`）— 主キー列の構成が変わり得るため、`canCreate`/`canDelete`の判定結果に影響する

## 3. アクセス権限のYAMLエクスポート／インポート（FR-2.10〜FR-2.12、PBT-02対象）

1. **エクスポート**（`PermissionYamlService.exportToYaml(connectionId)`）: 対象接続の現在の全権限設定を、プリンシパルを人間可読な識別子（ユーザは`email`、グループは`name`）で表現したYAMLとして出力する（Q6=A、BR-ACCESS-09）。エクスポート完了時、AuditEventPublisher経由で`PERMISSION_YAML_EXPORTED`イベントを記録する
2. **インポート**（`PermissionYamlService.importFromYaml(connectionId, yaml)`）:
   a. YAML内の各エントリについて、プリンシパル識別子（email／グループ名）を実際のユーザ／グループに解決する。解決できない識別子が1件でもあれば、インポート全体を拒否する（BR-ACCESS-09）
   b. 重複エントリ（同一プリンシパル×スキーマ×テーブル×カラムの主権限、または同一プリンシパル×スキーマ×テーブル×補助権限種別）が存在する場合、インポート全体を拒否する（FR-2.12、BR-ACCESS-10）
   c. 上記の検証をすべて通過した場合のみ、対象接続の既存権限設定をすべて削除し、YAMLの内容から再構築する（全置換、FR-2.11。単一トランザクション内で実施し、部分適用は行わない）
   d. インポート完了（成功／拒否）時、AuditEventPublisher経由で`PERMISSION_YAML_IMPORTED`イベントを記録する（拒否時は理由の概要を記録）
   e. インポート成功時、`invalidateCache()`を呼び出す

## 4. グループ管理（FR-2.13〜FR-2.15）

1. 管理者はグループを作成（`createGroup`）・改名（`renameGroup`）・削除（`deleteGroup`）できる
2. グループ削除時、当該グループの所属関係（`GroupMembership`）および当該グループに設定されている権限（`AccessPermission`、`principalType=GROUP`かつ`principalId=`当該グループ）をあわせてカスケード削除する（FR-2.13、BR-ACCESS-11）
3. 管理者はグループへのユーザ追加（`addUserToGroup`）・削除（`removeUserFromGroup`）を行える
4. グループ作成・改名・削除・所属追加・所属削除は、それぞれ別イベント種別としてAuditEventPublisher経由で記録する（Q7=A、BR-ACCESS-12）。いずれの操作も`invalidateCache()`を呼び出す（グループ構成の変更は実効権限に影響するため）
5. 権限設定（§1）は、ユーザ個別・グループのいずれのプリンシパルに対しても同じAPIで設定できる（FR-2.15、`principalType`区分による）

## 5. Testable Properties（PBT-01/02/03、Property-Based Testing拡張）

requirements.md §7.10のNFR-5.1で識別済みのPBT候補（権限判定・合成ロジック、YAML入出力）について、本ユニットで以下のプロパティを識別する。フレームワーク（jqwik、UNIT-02で確定済み）を用いた具体的なプロパティテスト実装はCode Generation段階で行う。

### 5.1 権限判定・合成ロジック（PBT-01: プロパティ識別、PBT-03: 不変条件）
- **階層優先の不変条件**: 同一プリンシパルの設定において、より詳細な階層（カラム）に設定が存在する限り、より粗い階層（テーブル／スキーマ）の設定値に関わらず、解決結果は常にカラム階層の値と一致する
- **個別設定優先の不変条件**: ユーザ自身の設定が対象リソースに1件でも存在する限り、所属グループの設定内容（どれだけ許可的であっても、あるいは制限的であっても）は解決結果に一切影響しない
- **グループ合成の単調性**: 各グループの権限設定のうち、いずれか1つでもより許可的な値（主権限:上位、補助権限:true）に変更すると、グループ合成結果は変化しないか、より許可的な方向にのみ変化する（決してより制限的にはならない）
- **作成可否判定の整合性**: `canCreate()`が`true`を返すのは、`CREATE`補助権限が`true`であり、かつ（主キーなし、または全主キー列の実効主権限が`UPDATE`）の場合に限られる（FR-2.6の式との同値性）
- **削除可否判定の整合性**: `canDelete()`が`true`を返すのは、`DELETE`補助権限が`true`であり、かつ主キーを持ち、かつ全主キー列の実効主権限が`READ`以上の場合に限られる。主キーなしテーブルでは常に`false`（FR-2.7の式との同値性）

### 5.2 YAML入出力（PBT-02: ラウンドトリップ特性）
- **ラウンドトリップ**: 重複のない任意の権限設定状態について、エクスポート→インポート→再エクスポートした結果は、最初のエクスポート結果と（プリンシパル・リソース・権限値の集合として）一致する
- **重複拒否の原子性**: 重複エントリを含む任意のYAMLをインポートしようとした場合、インポート全体が拒否され、対象接続の既存権限設定は一切変更されない
