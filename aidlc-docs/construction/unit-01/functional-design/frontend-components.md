# UNIT-01 デザインシステム基盤 - Frontend Components

`unit-01-functional-design-plan.md`の回答（Q1=A, Q2=A, Q3=A, Q4=B, Q5=A, Q6=A, Q7=A, Q8=A）に基づく。

UNIT-01はフロントエンドのみのユニットであり、Business Logic Modeling / Domain Model / Business Rules / Data Flow / Integration Pointsは対象外（N/A）。本ドキュメントはグランドデザイン（共通レイアウト）と代表5画面モックのコンポーネント構造のみを扱う。

## 1. グランドデザイン（共通レイアウト）

### 1.1 レイアウトの種類（Q1=A）

2種類の共通レイアウトを設ける。

| レイアウト | 適用画面 | 構成 |
|---|---|---|
| `PublicLayout` | ログイン、ユーザ登録（メール送信・PW設定） | ロゴ＋コンテンツのみの最小限シェル。ヘッダー/サイドナビ/フッターは持たない |
| `AppShell` | ログイン後の全画面（管理者ダッシュボード、マスタメンテナンス、権限設定 等） | ヘッダー／サイドナビ／コンテンツ領域／フッターで構成される共通シェル |

### 1.2 AppShellの構成要素

- **Header**: アプリ名/ロゴ、ログイン中ユーザ表示、ログアウト導線（プレースホルダー。実装はUNIT-02）
- **SideNav**: 折りたたみ可能な階層ナビゲーション。項目構成は1.3参照
- **Content**: 各画面のコンテンツをレンダリングする領域
- **Footer**: バージョン情報等のプレースホルダー

### 1.3 ナビゲーション項目（Q2=A: 全10ユニットを見込んだ仮決め）

実装予定の全機能を見込んだナビ項目一式を仮決めする。リンク先は各ユニット未着手の間はプレースホルダー（実装済みユニットのみ実際に遷移可能）。

| ナビ項目 | 対応ユニット |
|---|---|
| ダッシュボード（承認待ちユーザ一覧） | UNIT-02 |
| ユーザ管理 | UNIT-02 |
| RDBMS接続設定 | UNIT-03 |
| アクセス制御（権限設定・グループ管理） | UNIT-04 |
| マスタメンテナンス | UNIT-05 |
| 保存クエリ | UNIT-06 |
| クエリビルダー | UNIT-07 |
| クエリ実行履歴 | UNIT-08 |
| 監査ログ | UNIT-09 |

### 1.4 レスポンシブ対応（Q5=A）

デスクトップ優先。管理者・DB操作者向けの業務ツールという性質上、モバイル対応は将来検討とする（本ユニットではブレークポイント設計を行わない）。

## 2. 代表画面モックの配置・性質（共通事項）

- **配置場所（Q7=A）**: 共通シェル・トークン等は`frontend/src/design-system/`配下に集約し、代表画面モックは`frontend/src/mocks/`配下に、本実装コードと明確に区別して配置する（各ユニット本実装時は`frontend/src/pages/`等に別途実装し、モックのコードをそのまま転用しない）
- **モックデータ（Q3=A）**: それらしい日本語のダミーデータ（実データ風サンプル）を使用する
- **操作性（Q6=A）**: 見た目のみの静的モック。入力・送信・API連携等の実際の動作は持たない
- **画面状態（Q4=B）**: 各画面について、通常状態に加え、空状態・エラー状態等の主要なバリエーションも用意する

## 3. 代表5画面のコンポーネント構造

### 3.1 ログイン画面（`PublicLayout`）

- **構成**: `AuthCard`（ロゴ＋タイトル）> `TextField`（メールアドレス）, `TextField`（パスワード、type=password）, `Button`（プライマリ、「ログイン」）
- **画面状態**:
  - 通常（未入力）
  - エラー（認証失敗メッセージを`ErrorAlert`で表示）

### 3.2 ユーザ登録画面（メール送信・PW設定）（`PublicLayout`）

2段階登録フロー（FR-1.1〜1.3）に対応する2つの画面モック。

- **Step 1（メールアドレス送信）**: `TextField`（メールアドレス）, `Button`（「送信」）
  - 通常（未入力）
  - 送信完了（`SuccessAlert`で「確認メールを送信しました」を表示）
- **Step 2（パスワード設定、メール内リンクから遷移）**: `TextField`（パスワード）, `TextField`（パスワード確認）, `Button`（「登録完了」）
  - 通常（未入力）
  - エラー（トークン期限切れ・無効なトークンを`ErrorAlert`で表示）

### 3.3 管理者ダッシュボード（承認待ちユーザ一覧）（`AppShell`）

- **構成**: `PageHeader`（タイトル）> `DataTable`（列: メールアドレス、登録日時、アクション［承認／却下ボタン］）, `ConfirmDialog`（承認／却下の確認）
- **画面状態**:
  - 通常（承認待ちユーザのダミーデータを複数件表示）
  - 空状態（`EmptyState`で「承認待ちユーザはいません」を表示）
  - エラー（一覧取得失敗を`ErrorAlert`で表示）

### 3.4 マスタメンテナンス画面（一覧・編集）（`AppShell`）

- **構成**: `ConnectionSelector` / `TableSelector`（対象RDBMS接続・テーブル選択）, `FilterBar`（絞り込み）, `DataTable`（レコード一覧、ページネーション付き）, `RecordEditForm`（モーダルでのレコード編集）, `Button`（「保存」）
- **画面状態**:
  - 通常（ダミーレコードを複数件表示）
  - 空状態（`EmptyState`で「レコードがありません」を表示）
  - エラー（接続失敗・権限なしを`ErrorAlert`で表示）

### 3.5 権限設定画面（`AppShell`）

Q8=Aにより、UNIT-04で正式設計する権限モデル（主権限・補助権限）は先取りせず、一般的な設定画面レイアウトとして表現する。

- **構成**: `PermissionTable`（ユーザ／グループ×リソースの一覧）, `EditForm`（権限選択のプルダウン・チェックボックス群、汎用的なプレースホルダー項目として表現）, `Button`（「保存」）
- **画面状態**:
  - 通常（ダミーの権限設定を複数件表示）
  - 空状態（`EmptyState`で「権限が設定されていません」を表示）
  - エラー（保存失敗を`ErrorAlert`で表示）

## 4. UNIT-01で新設する共通コンポーネント（代表画面モックから導出）

上記の代表画面モックを構築する過程で、既存の基本部品（ボタン・フォーム部品等）に加え、以下の共通コンポーネントが必要になる。UNIT-01のCode Generationで実装する。

- `PublicLayout` / `AppShell` / `Header` / `SideNav` / `Footer`
- `AuthCard`
- `PageHeader`
- `DataTable`（列定義・ページネーション対応）
- `EmptyState`
- `ErrorAlert` / `SuccessAlert`
- `ConfirmDialog`
- `FilterBar`
