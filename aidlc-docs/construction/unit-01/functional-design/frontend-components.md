# UNIT-01 デザインシステム基盤 - Frontend Components

`unit-01-functional-design-plan.md`の回答（Q1=A, Q2=A, Q3=A, Q4=B, Q5=A, Q6=A, Q7=A, Q8=A）に基づく。

UNIT-01はフロントエンドのみのユニットであり、Business Logic Modeling / Domain Model / Business Rules / Data Flow / Integration Pointsは対象外（N/A）。本ドキュメントはグランドデザイン（共通レイアウト）と代表5画面モックのコンポーネント構造のみを扱う。

## 1. グランドデザイン（共通レイアウト）

### 1.1 レイアウトの種類（Q1=A）

2種類の共通レイアウトを設ける。

| レイアウト | 適用画面 | 構成 |
|---|---|---|
| `PublicLayout` | ログイン、ユーザ登録（メール送信・PW設定） | ロゴ＋コンテンツのみの最小限シェル。ヘッダー/サイドナビ/フッターは持たない |
| `AppShell` | ログイン後の全画面（管理者ダッシュボード、マスタメンテナンス、権限設定 等） | ヘッダー／サイドナビ／コンテンツ領域／フッターで構成される共通シェル |

### 1.2 AppShellの構成要素

- **Header**: アプリ名/ロゴ、ログイン中ユーザ表示、ログアウト導線（プレースホルダー。実装はUNIT-02）
- **SideNav**: 折りたたみ可能な階層ナビゲーション。項目構成は1.3参照
- **Content**: 各画面のコンテンツをレンダリングする領域
- **Footer**: バージョン情報等のプレースホルダー

### 1.3 ナビゲーション項目（Q2=A: 全10ユニットを見込んだ仮決め）

実装予定の全機能を見込んだナビ項目一式を仮決めする。リンク先は各ユニット未着手の間はプレースホルダー（実装済みユニットのみ実際に遷移可能）。

| ナビ項目 | 対応ユニット |
|---|---|
| ユーザ管理（承認待ちユーザ一覧を含む） | UNIT-02 |
| RDBMS接続設定 | UNIT-03 |
| アクセス制御（権限設定・グループ管理） | UNIT-04 |
| マスタメンテナンス | UNIT-05 |
| 保存クエリ | UNIT-06 |
| クエリビルダー | UNIT-07 |
| クエリ実行履歴 | UNIT-08 |
| 監査ログ | UNIT-09 |

**訂正（UNIT-02 Functional Designにて）**: 当初「ダッシュボード（承認待ちユーザ一覧）」と「ユーザ管理」を別ナビ項目として仮決めしていたが、両者は対象データ・操作が大きく重複するため1画面「ユーザ管理」に統合した。また、統合により「承認待ちユーザ一覧（管理者専用）」がログイン後の実質的な着地点になっていたが、これは一般ユーザにとっての着地点として機能しないため、別途ログイン後のトップ画面（ホーム、SideNav項目ではなくAppShellのルート`/`。上記ナビ項目一覧へのカードリンクを配置）をUNIT-02で新設した。詳細はaidlc-docs/construction/unit-02/functional-design/frontend-components.md参照

### 1.4 レスポンシブ対応（Q5=A、requirements.md §7.9 NFR-8.1に準拠するよう修正）

デスクトップ利用を中心とするが、NFR-8.1「タブレット等でも大きく崩れない程度のレスポンシブ対応を行う（モバイル本格対応は不要）」に従い、以下を最低限考慮する。

- タブレット幅（目安: 768px前後）でSideNavが折りたたみ表示に切り替わる等、レイアウトが大きく崩れないブレークポイントを1つ設ける
- モバイル幅（スマートフォン）向けの専用レイアウト最適化は本ユニットの対象外とする（将来検討）

## 2. 代表画面モックの配置・性質（共通事項）

- **配置場所（Q7=A）**: 共通シェル・トークン等は`frontend/src/design-system/`配下に集約し、代表画面モックは`frontend/src/mocks/`配下に、本実装コードと明確に区別して配置する（各ユニット本実装時は`frontend/src/pages/`等に別途実装し、モックのコードをそのまま転用しない）
- **モックデータ（Q3=A）**: それらしい日本語のダミーデータ（実データ風サンプル）を使用する
- **操作性（Q6=A）**: 見た目のみの静的モック。入力・送信・API連携等の実際の動作は持たない
- **画面状態（Q4=B）**: 各画面について、通常状態に加え、空状態・エラー状態等の主要なバリエーションも用意する
- **閲覧経路**: モックは開発ビルド限定のルート`/mock/*`配下でのみアクセス可能とし、本番ビルドには含めない。パス割り当ては以下のとおり確定する

| パス | 内容 |
|---|---|
| `/mock/catalog` | コンポーネントカタログ（共通コンポーネント一覧） |
| `/mock/login` | ログイン画面モック |
| `/mock/register` | ユーザ登録画面モック（メール送信・PW設定の2ステップ） |
| `/mock/dashboard` | 管理者ダッシュボードモック |
| `/mock/master-data` | マスタメンテナンス画面モック |
| `/mock/permissions` | 権限設定画面モック |
- **コンポーネントの実装レベル**: `DataTable`/`FilterBar`/`ConfirmDialog`/`PermissionTable`等は、本ユニットでは静的モック表示に必要な最小限（見た目のみ、ソート・ページング等の実ロジックなし）の簡易版として実装する。実データ対応（ソート・ページング・フィルタリングロジック等）は、各機能を実装する後続ユニット（UNIT-04、UNIT-05等）で必要に応じて拡張する
- **アイコン**: サードパーティのアイコンライブラリは使用せず、必要なアイコンはSVGで自作する（FR-0.1のサードパーティUIライブラリ不使用方針に準拠）

## 3. 代表5画面のコンポーネント構造

### 3.1 ログイン画面（`PublicLayout`）

- **構成**: `AuthCard`（ロゴ＋タイトル）> `TextField`（メールアドレス）, `TextField`（パスワード、type=password）, `Button`（プライマリ、「ログイン」）
- **画面状態**:
  - 通常（未入力）
  - エラー（認証失敗メッセージを`ErrorAlert`で表示）

### 3.2 ユーザ登録画面（メール送信・PW設定）（`PublicLayout`）

2段階登録フロー（FR-1.1〜1.3）に対応する2つの画面モック。

- **Step 1（メールアドレス送信）**: `TextField`（メールアドレス）, `Button`（「送信」）
  - 通常（未入力）
  - 送信完了（`SuccessAlert`で「確認メールを送信しました」を表示）
- **Step 2（パスワード設定、メール内リンクから遷移）**: `TextField`（パスワード）, `TextField`（パスワード確認）, `Button`（「登録完了」）
  - 通常（未入力）
  - エラー（トークン期限切れ・無効なトークンを`ErrorAlert`で表示）

### 3.3 管理者ダッシュボード（承認待ちユーザ一覧）（`AppShell`）

- **構成**: `PageHeader`（タイトル）> `DataTable`（列: メールアドレス、登録日時、アクション［承認／却下ボタン］）, `ConfirmDialog`（承認／却下の確認）
- **画面状態**:
  - 通常（承認待ちユーザのダミーデータを複数件表示）
  - 空状態（`EmptyState`で「承認待ちユーザはいません」を表示）
  - エラー（一覧取得失敗を`ErrorAlert`で表示）

**訂正（UNIT-02 Functional Designにて）**: このモック（`/mock/dashboard`）が想定していた「承認待ちユーザ専用の管理者ダッシュボード」は、UNIT-02の本実装では全ステータス対象の「ユーザ管理」画面に統合された（承認/却下アクションもこの画面に含まれる）。FR-0.5のとおりモックの見た目をそのまま踏襲するとは限らないため、モックコード自体（`frontend/src/mocks/`）は変更していない。

### 3.4 マスタメンテナンス画面（一覧・編集）（`AppShell`）

- **構成**: `ConnectionSelector` / `TableSelector`（対象RDBMS接続・テーブル選択）, `FilterBar`（絞り込み）, `DataTable`（レコード一覧、ページネーション付き）, `RecordEditForm`（モーダルでのレコード編集）, `Button`（「保存」）
- **画面状態**:
  - 通常（ダミーレコードを複数件表示）
  - 空状態（`EmptyState`で「レコードがありません」を表示）
  - エラー（接続失敗・権限なしを`ErrorAlert`で表示）

### 3.5 権限設定画面（`AppShell`）

Q8=Aにより、UNIT-04で正式設計する権限モデル（主権限・補助権限）は先取りせず、一般的な設定画面レイアウトとして表現する。

- **構成**: `PermissionTable`（ユーザ／グループ×リソースの一覧）, `EditForm`（権限選択のプルダウン・チェックボックス群、汎用的なプレースホルダー項目として表現）, `Button`（「保存」）
- **画面状態**:
  - 通常（ダミーの権限設定を複数件表示）
  - 空状態（`EmptyState`で「権限が設定されていません」を表示）
  - エラー（保存失敗を`ErrorAlert`で表示）

## 4. UNIT-01で新設する共通コンポーネント（代表画面モックから導出、参考実装の取込方針を反映）

`reference/design-system/`との突き合わせにより、以下を確定する（Code Generation計画作成時の決定）。取り込み範囲はユーザー判断により拡大し、参考実装のコンポーネント一式を今回まとめて構築する。

**基本部品・フォーム**
- `Button`, `TextInput`, `Choice`（`Checkbox` / `RadioGroup` / `Switch`）
- `FormField`（`cloneElement`方式でid/aria属性を子要素へ注入。参考実装と同じパターン）

**グランドデザイン**
- `PublicLayout` / `AppShell` / `Header` / `SideNav` / `Footer`
- `HeaderControl`（ユーザメニュー等のヘッダー内コントロール群）
- `LanguageSwitcher`（i18n言語切替）
- `ThemeToggle`（ダークモード切替）

**表示・フィードバック**
- `AuthCard`, `Card`（汎用カード。`AuthCard`は`Card`を土台に構築）
- `PageHeader`
- `DataTable`（`Table`を土台に、列定義・簡易表示のみ実装。ソート・ページングの実ロジックは後続ユニットで拡張）
- `Pagination`（見た目のみ、ページ送りの実ロジックは後続ユニットで拡張）
- `EmptyState`
- `Alert`（`tone: info/success/warning/danger`の単一コンポーネント。当初案の`ErrorAlert`/`SuccessAlert`を統合）
- `Badge`
- `Spinner`
- `Modal`（汎用の`Overlay`背景幕コンポーネントは存在せず、Modal内部で直接処理する）
- `ConfirmDialog`（`Modal`を土台に構築）
- `FilterBar`（簡易版）
- `Tabs`
- `Toast`
- `CodeBlock`
- `KeyValueList`
- `Dropdown`（HeaderControlのユーザーメニュー等で使用。キーボード操作・フォーカストラップ対応）
- `Tooltip`（ホバー/フォーカスで表示する補足情報）
- `Icon`（自作SVGアイコンセット）

**対象外**: `reference/mocks/pages/screens/QueryRunMock.tsx`はUNIT-01の代表5画面（FR-0.4）に含まれないため、画面モックとしては今回作成しない（クエリ実行はUNIT-06/07のスコープ）。

## 5. Testable Properties（PBT-01対応）

UNIT-01は静的な見た目のみの提示用コンポーネント・モックであり、往復変換・不変条件・冪等性・可換性・オラクル・帰納法・検証容易性のいずれのプロパティも識別されない。**No PBT properties identified** — アルゴリズム的処理・データ変換を一切含まないため（stories.md STORY-0.1〜0.3も「PBT対象: いいえ」と明記済み）。
